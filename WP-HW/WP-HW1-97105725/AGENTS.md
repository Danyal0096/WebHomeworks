# AGENTS.md — Web Programming HW1: Notion-Like Notes App

## 1. Mission

Build a polished, fully working, client-side notes application inspired by Notion for a Web Programming homework. The application must be implemented with **plain HTML, CSS, and JavaScript only**.

This is not a prototype. Deliver a complete static application that can be opened in a modern browser and used without a server, packages, libraries, frameworks, build tools, CDNs, or external dependencies.

The UI must support Persian and English. Persian is the default UI language and uses RTL layout; English uses LTR layout. User-created note and folder data must never be translated or modified when the UI language changes.

Do not ask for clarification unless the repository contains a direct conflict with these instructions. Inspect the repository first, then implement the project completely.

---

## 2. Hard Constraints — Do Not Violate

1. **No third-party packages, libraries, frameworks, or external assets.**
   - No React, Vue, jQuery, Bootstrap, Tailwind, Markdown libraries, icon libraries, state libraries, or date libraries.
   - No CDN scripts, CDN stylesheets, web-font imports, package managers, or build setup.
   - Use browser-native APIs only.

2. **Do not use JavaScript modules.**
   - Do not use `type="module"`.
   - Do not use `import` or `export`.
   - Use multiple classic JavaScript files loaded with `defer` in a deliberate dependency order.

3. **Use multiple files.**
   - Keep HTML, CSS, and JavaScript separated.
   - Keep responsibilities separated. Do not place the entire application in one giant script file.

4. **No backend.**
   - Persist application data with `localStorage`.
   - Import/export through browser file APIs only.

5. **No fake features.**
   - Every visible action must work.
   - No nonfunctional toolbar controls, dummy buttons, TODO placeholders, or mocked persistence.

6. **Avoid unsafe rendering.**
   - User content must not be inserted as arbitrary unsanitized HTML.
   - The Markdown renderer may support only the explicitly allowed formatting tags generated by this application; it must escape all other raw HTML.

---

## 3. Required Product Behavior

Create a simple Notion-like notes app with a persistent sidebar and a main note area.

### Core requirement summary

- Create, edit, delete, view, search, sort, pin, and move notes.
- Create, rename, delete, collapse, and expand folders.
- Folders have **only one level of depth**. Folder-inside-folder is forbidden.
- Persist all relevant data and preferences in `localStorage`.
- Export all stored app data to a JSON file.
- Import a valid JSON file only after confirmation; import **replaces** current persisted app data, it never merges.
- Support simplified Markdown rendering in view mode.
- Use raw Markdown editing in edit mode.
- Support dark/light theme and Persian/English UI switching.
- Use responsive layout without hiding the sidebar behind a drawer.

## 4. UX and Layout Specification

### 4.1 Application shell

The app has two permanent regions:

1. **Sidebar**
   - In Persian/RTL, it appears on the right.
   - In English/LTR, it appears on the left.
   - On desktop it remains visible and sticky while using the app.
   - On small screens it must remain available and visible as a compact top section; do not convert it into a hidden drawer.

2. **Main area**
   - Shows a useful empty state when no note is selected.
   - Shows one selected note in either view mode or edit mode.

Use Flexbox/Grid and CSS logical properties so changing `dir` changes the side naturally. Avoid hard-coded physical properties such as `left`, `right`, `margin-left`, `padding-right`, etc., except where truly unavoidable. Prefer `margin-inline-*`, `padding-inline-*`, `border-inline-*`, `inset-inline-*`, and `text-align: start`.

### 4.2 Sidebar content

The sidebar must contain:

- App title/identity.
- Search input.
- A note creation button.
- A folder creation button.
- A sort control.
- A **Pinned notes** section.
- A **Folders** section.
- An **Unfiled/All notes** section for notes without a folder.
- Import and export controls.
- Theme toggle.
- Language toggle.

Pinned notes are shown at the top as quick-access entries and have a visible pin marker. They may also remain visible in their normal folder/unfiled position; the pinned section is a shortcut, not a relocation.

When search is active:

- Render one flat, non-duplicated search-results list.
- Search against raw note `title` and `content`.
- Keep result ordering consistent with the selected sort mode.
- Show a clear empty-result message if nothing matches.

Folder behavior:

- Clicking a folder name only collapses/expands its note list.
- A folder has no separate document page.
- Rename/delete actions are available through a compact action menu or accessible icon buttons.
- Folder deletion requires confirmation.
- If confirmed, delete the folder **and all notes inside it**.
- If the current selected note is deleted because its folder was deleted, clear selection and show the empty state.

### 4.3 Main area states

#### Empty state

When no note is selected, show a concise empty state with a localized button that creates a new note.

#### View mode

When a note is selected and not being edited:

- Render the note title.
- Render formatted Markdown content.
- Show clear actions: edit, pin/unpin, delete.
- Optionally show compact created/updated metadata.
- The content surface should be readable and visually polished.

#### Edit mode

When editing a selected note:

- Show a title input.
- Show an editing toolbar.
- Show a textarea containing the **raw** saved Markdown/formatting source.
- Provide a `Save & Exit` action. This action exits edit mode; it does not need to be the only persistence trigger because auto-save happens continuously.
- Provide a folder selector so the note can be moved to any folder or to `No folder`.
- Preserve edits if the user selects another note; changes must already have been auto-saved.

Do **not** show a permanent second preview panel or a separate Markdown-preview button. The app has exactly two note modes:

- View mode: rendered Markdown.
- Edit mode: raw Markdown source.

## 5. Data Model and Persistence

Use one normalized root application state. Keep IDs stable and use ISO timestamps.

Use this shape (property naming may remain exactly as shown):

```js
{
  schemaVersion: 1,
  settings: {
    locale: "fa",        // "fa" | "en"
    theme: "light",      // "light" | "dark"
    sortMode: "updated-desc" // see supported modes below
  },
  folders: [
    {
      id: "folder-...",
      name: "...",
      createdAt: "2026-...",
      updatedAt: "2026-..."
    }
  ],
  notes: [
    {
      id: "note-...",
      title: "...",
      content: "...",      // raw Markdown + allowed app-generated inline tags
      folderId: null,        // string | null
      isPinned: false,
      createdAt: "2026-...",
      updatedAt: "2026-..."
    }
  ],
  ui: {
    selectedNoteId: null,
    collapsedFolderIds: []
  }
}
```

Rules:

- `folderId` must either be `null` or refer to an existing folder.
- Do not allow nested folders in data or UI.
- `isPinned` is always a boolean.
- Update `updatedAt` whenever a note/folder changes.
- Use a browser-safe unique ID approach such as `crypto.randomUUID()` with a safe fallback.
- Keep transient UI details such as the current search text and current edit mode out of exported core data unless there is a strong reason to store them.

### Persistence rules

- Use one documented localStorage key, for example `notionLite.state.v1`.
- Load the state safely on startup.
- If no valid state exists, start with an empty state.
- All mutations must go through a central store/service that:
  1. validates or normalizes state as needed,
  2. updates in-memory state,
  3. saves to localStorage,
  4. notifies the UI to re-render.
- Do not scatter direct `localStorage` reads/writes across controllers or view files.

## 6. Notes, Folders, Pinning, Search, and Sort

### Notes

Implement all of the following:

- Create a note.
  - Create it with a localized default title such as `یادداشت بدون عنوان` / `Untitled note`.
  - Select it immediately.
  - Enter edit mode immediately.
- Update title and content.
- Delete a note after confirmation.
- Move a note between folders with the edit-mode folder selector.
- Pin/unpin a note.
- View a note by clicking it in the sidebar.

### Folders

Implement:

- Create folder through a localized modal/input.
- Rename folder through a localized modal/input.
- Delete folder after confirmation.
- On confirmed folder deletion, delete every note whose `folderId` matches that folder.
- Folder lists are collapsible.

### Pinning

- Any number of notes can be pinned.
- Pinned notes appear in the pinned section at the top of the sidebar.
- Show a visible pin icon/marker in every appropriate location.
- Pin/unpin must persist immediately.

### Search

- Search case-insensitively against note title and raw content.
- Search must update while typing.
- Use a localized no-results state.

### Sort

Provide a localized sort selector with at least these modes:

- `title-asc` — alphabetical title order.
- `created-desc` — newest created first.
- `updated-desc` — most recently updated first.

Use `Intl.Collator` when sorting titles so Persian and English ordering are handled reasonably.

## 7. Edit Toolbar and Formatting Storage

The toolbar is not decorative. It must change the note source text so formatting survives reload, localStorage, JSON export, and JSON import.

Use a `<textarea>` for raw Markdown editing. Do not use `contenteditable` as the primary stored editor format.

Toolbar behavior on selected text:

- Bold → wrap selection as `**selected text**`
- Italic → wrap selection as `*selected text*`
- Underline → wrap selection as `<u>selected text</u>`
- Text color → wrap selection as `<span style="color: #RRGGBB">selected text</span>`

Use a browser-native color input (`<input type="color">`) for arbitrary color selection; do not use a library.

Implementation details that matter:

- Preserve the textarea selection range before toolbar interaction.
- Toolbar clicks must use the current/last valid selected range even if the button temporarily steals focus.
- If no text is selected, either disable the formatting action or show a localized, non-disruptive hint. Do not silently corrupt content.
- Update content, `updatedAt`, localStorage, and UI state immediately after a formatting action.

The editor may include a small localized Markdown helper/reference, but it must not turn the UI into a second preview block.

## 8. Simplified Markdown Support

Implement a small, dependency-free Markdown renderer. It must support the following:

### Block syntax

- Headings: `# Heading`, `## Heading`, through reasonable heading levels.
- Unordered lists: `- item` and/or `* item`.
- Ordered lists: `1. item`.
- Fenced code blocks with triple backticks.
- Paragraphs.

### Inline syntax

- Bold: `**text**` and `__text__`.
- Italic: `*text*` and `_text_`.
- Links: `[text](url)`.
- Inline code: `` `code` ``.
- Underline: `<u>text</u>`.
- Color: `<span style="color: #RRGGBB">text</span>`.

### Security and implementation expectations

- The renderer must escape arbitrary HTML first.
- It may restore only the narrow allowlist of app-supported underline and color tags after validating them.
- Do not permit arbitrary attributes, arbitrary inline styles, scripts, event handlers, iframes, or arbitrary HTML tags.
- Code block and inline code content must be escaped and must not be interpreted as Markdown inside the code region.
- For links, allow safe protocols such as `http:`, `https:`, and `mailto:`. Reject or render as plain text any unsafe URL such as `javascript:`.
- Open external links with `target="_blank"` and `rel="noopener noreferrer"`.
- Do not claim full CommonMark compatibility. The goal is correct support for the required simplified syntax.

Keep Markdown parsing isolated in a dedicated service. Do not mix it into the main UI rendering code.

## 9. Import and Export JSON

### Export

Export all user data and user preferences required to restore the application:

- schema version,
- settings,
- folders,
- notes,
- persistent UI preferences such as collapsed folders/selected note if included in state.

Use `Blob`, `URL.createObjectURL`, and an anchor with `download` to trigger a browser download. Use a meaningful filename such as:

```text
notion-lite-export-YYYY-MM-DD.json
```

The browser controls the physical save location.

### Import

Implement import through a file picker accepting JSON.

Strict import flow:

1. Read file content.
2. Parse JSON safely.
3. Validate the full expected schema before changing any current state.
4. If invalid, show a localized error and leave existing state untouched.
5. If valid, show a localized confirmation modal explaining that current application data will be replaced.
6. Only after confirmation, replace current core state entirely.
7. Persist the imported state immediately to localStorage.
8. Re-render the app and reset transient edit/search state safely.

Never merge imported data with current data.

Validation should reject clearly malformed data, including at least:

- wrong root shape,
- missing arrays for folders/notes,
- duplicate IDs,
- invalid settings values,
- notes referencing nonexistent folders,
- invalid timestamp/value types,
- invalid `isPinned` values.

Keep import/export/validation responsibilities separate from rendering.

## 10. Internationalization, RTL/LTR, and Theme

### Language

Implement UI localization with a central translation dictionary, not scattered string literals.

Supported locales:

- Persian: `fa` (default)
- English: `en`

Changing language must:

- update all visible UI labels, placeholders, modal texts, button labels, status text, and empty states;
- set `document.documentElement.lang` correctly;
- set `document.documentElement.dir` to `rtl` for Persian and `ltr` for English;
- update layout naturally through CSS logical properties;
- persist the language setting;
- not translate, mutate, reorder, or otherwise alter user note/folder data.

Include translations for every visible string, including confirm/error/success messages.

### Theme

Implement light and dark themes using CSS custom properties.

- Persist selected theme.
- Use a `data-theme` attribute on the root element.
- Respect system preference only as an optional initial fallback when no persisted preference exists.
- Ensure adequate contrast in both themes.

## 11. Responsive and Accessible UI

The interface must be pleasant and usable, not merely technically complete.

Required UI quality:

- A clean, simple Notion-like visual hierarchy.
- Strong spacing, readable typography, and visible selected/hover/focus states.
- Responsive behavior for desktop and narrow mobile viewports.
- Sidebar remains accessible at every viewport; never hide it behind an inaccessible drawer.
- Main content and sidebar should scroll independently where appropriate.
- Use semantic HTML where possible: `aside`, `main`, `nav`, `button`, `label`, `input`, `textarea`, etc.
- Every icon-only control needs an accessible localized label (`aria-label`/tooltip as appropriate).
- Focus styles must remain visible.
- Use localized confirmation modal(s) for destructive actions and import replacement.
- Avoid browser alert spam. Use a small reusable toast/status component for normal feedback.

Do not use external fonts or icons. Use text labels, Unicode characters, CSS, or simple inline SVG created locally if needed.

## 12. Architecture and SOLID Expectations

Use clean, pragmatic vanilla JavaScript architecture. Do not overengineer the app into a fake enterprise framework, but do not create one giant file either.

### Required file structure

Create or adapt to a structure close to this:

```text
index.html
css/
  reset.css
  tokens.css
  layout.css
  components.css
  responsive.css
js/
  core/
    namespace.js
    constants.js
    store.js
  services/
    id-service.js
    storage-service.js
    validation-service.js
    note-service.js
    folder-service.js
    markdown-service.js
    i18n-service.js
    transfer-service.js
  ui/
    dom.js
    sidebar-view.js
    main-view.js
    editor-controller.js
    modal.js
    toast.js
  app.js
```

You may make small structural improvements if they preserve the same separation of responsibilities.

### Classic-script loading pattern

- Put `defer` script tags in `index.html` in dependency order.
- Create a single minimal global namespace, for example:

```js
window.NotionLite = window.NotionLite || {};
```

- Each classic JS file attaches only its narrow public API to that namespace.
- Do not create unrelated globals.
- Do not use ES module syntax.

### Responsibility boundaries

- **Store**: owns state, mutation, subscriptions, and persistence trigger.
- **Storage service**: only reads/writes localStorage safely.
- **Validation service**: validates imported state/data.
- **Note service**: note-specific operations.
- **Folder service**: folder-specific operations.
- **Markdown service**: parsing/rendering only.
- **Transfer service**: JSON export/import file handling only.
- **I18n service**: translations, locale switching, document `dir/lang` only.
- **UI views/controllers**: render DOM and bind events; do not own business rules.
- **App bootstrap**: wires services once and initializes the application.

### Design rules

- Use JSDoc typedefs for core state shapes and public function contracts.
- Avoid duplicated logic.
- Avoid direct store mutation from arbitrary UI code.
- Use small pure helper functions where useful.
- Keep DOM querying centralized or scoped to the component/view that owns it.
- Re-render safely after state changes without duplicating event handlers.
- Prefer event delegation for dynamic sidebar lists where useful.
- Do not use `eval`, `new Function`, or unsafe HTML injection.

## 15. Completion Standard

When finished, leave the repository with a working application rather than a design document. Keep implementation files coherent, avoid unnecessary complexity, and ensure the app can run directly as a static website.

Prioritize this ordering when tradeoffs arise:

1. Correct, complete functionality.
2. Compliance with the no-library/no-module constraints.
3. Data safety and persistence correctness.
4. Clear architecture and maintainability.
5. Visual polish and responsive behavior.

Do not sacrifice functionality merely to make the code look architecturally elegant.
